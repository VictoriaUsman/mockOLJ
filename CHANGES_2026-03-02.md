Date: March 2, 2026
Project: Centralized Property Management Database
Session: Schema v6 — Gmail Loader Alignment

What I did today:
Reviewed the Gmail loader output spec (output/domains/gmail.json) against the existing gmail_threads table in v5. Found 3 columns the loader populates for every thread that had no place to land in the database. Wrote migrate_v6.sql to add them and updated schema.sql so fresh builds match. gmail_emails was already complete — no changes needed there.

Gaps found and fixed:

- gmail_threads.delegate_email — which service-account mailbox the thread was fetched from (noah@, hello@, bookkeeping@, evergreencottages@staywithprecision.com). The loader fetches all four and tags each thread; without this column that information was silently dropped.
- gmail_threads.message_count — number of messages in the thread at sync time. Used by the loader to populate the threads section of gmail.json.
- gmail_threads.history_id — Gmail's historyId field, needed for incremental / delta sync so the loader knows where to resume on the next run.

Also added idx_gmail_threads_delegate index on delegate_email to support filtering threads by mailbox.

Files changed:
- migrate_v6.sql (new) — run this once against the live database to apply all changes
- schema.sql — updated in place (gmail_threads definition + delegate index + version bumped to 6)

What it means for the team:
When the Gmail loader writes its output into the database, every field in the threads section of gmail.json now has a column. The delegate_email column in particular lets you filter or group threads by mailbox, which is important now that we're ingesting from four separate accounts. history_id sets up incremental sync so future runs don't have to re-fetch everything from scratch.
